# DOC-126 – PDF Generation Specification

**Document ID:** DOC-126
**Version:** 1.1
**Status:** Draft
**Owner:** BobKat IT
**Last Updated:** June 25, 2026

---

# 1. Purpose

DOC-126 defines the architecture, standards, branding, formatting, and generation rules for all PDF documents produced by StackScore.

The objective is to ensure every generated document reflects the professionalism of BobKat IT while communicating business value clearly and consistently.

DOC-126 is a **PDF generation architecture specification only**. It does not define rendering code, template files, or application logic.

**Relationship to other documents:**

* [DOC-125 – Reporting Engine Specification](DOC-125%20%E2%80%93%20Reporting%20Engine%20Specification.md) defines **what** reports exist and their data rules.
* [DOC-005 – UI & UX Standards](../00-Governance/DOC-005%20%E2%80%93%20UI%20&%20UX%20Standards.md) defines canonical typography, color, and visual identity shared with the application.
* DOC-126 defines **how** approved report content is presented in PDF form.

---

# 2. PDF Generation Philosophy

* Every PDF represents the **BobKat IT brand**.
* PDFs communicate **business outcomes** before technical details.
* **Executive readability** is the highest priority.
* **Visual consistency** exists across every generated document.
* Reports feel like **professional consulting deliverables**, not software exports.

PDF generation is invoked by the Reporting Service and Document Service per [DOC-124 – Service Layer Specification](DOC-124%20%E2%80%93%20Service%20Layer%20Specification.md) after content has been authorized and redacted per [DOC-122 – Roles & Permissions Specification](DOC-122%20%E2%80%93%20Roles%20&%20Permissions%20Specification.md).

---

# 3. Design Principles

1. **One visual system** — all PDFs share typography, color, spacing, and component styles from DOC-005.
2. **Progressive depth** — executive summary first; detail follows in supporting sections.
3. **Whitespace and hierarchy** — avoid dense technical dumps; use clear headings and section breaks.
4. **Data with context** — every chart includes a one-sentence business interpretation.
5. **Print-first** — layouts work on US Letter (primary) and A4 (optional future).
6. **Role-aware output** — same report type may produce redacted vs full PDF variants.
7. **Immutable deliverables** — delivered client PDFs are version-stamped and archived.

---

# 4. Standard Branding

Every PDF shall include:

| Element | Requirement |
| ------- | ------------- |
| BobKat IT logo | Header or cover — approved asset only |
| StackScore branding | Product mark or "Generated by StackScore" attribution |
| Document title | Primary title matching report type |
| Client name | Prominent on cover and header/footer |
| Consultant | Name of lead consultant or "BobKat IT" |
| Assessment or generation date | Date of source assessment or PDF generation |
| Document version | Version label when applicable (see Section 13) |
| Confidential footer | Standard confidentiality line on every page |
| Page numbers | "Page X of Y" footer |

**Color and typography** shall align with DOC-005 tokens. PDFs do not introduce ad-hoc fonts or colors per report.

**Confidential footer (default):**

> Confidential — Prepared for [Client Name] by BobKat IT. Unauthorized distribution prohibited.

Internal-only PDFs use:

> Internal Use Only — BobKat IT. Not for client distribution.

---

# 5. Cover Page Standards

Every **client-facing** PDF shall include a cover page with:

* Client name
* Document title
* Executive subtitle (one line — business outcome framing)
* Generated by StackScore
* BobKat IT branding (logo + wordmark)
* Generation date
* Document version (when versioned)
* Confidential designation where applicable
* Optional: consultant name and BobKat IT contact

**Cover page must not include:** internal costs, playbook names, or draft watermarks on delivered documents.

**Internal-only PDFs** may use a simplified cover with "Internal Use Only" banner.

---

# 6. Executive Summary Standards

Every client-facing strategic PDF shall open with an executive summary (after cover) that answers:

| Question | Content guidance |
| -------- | ---------------- |
| **Where are we today?** | Current StackScore, maturity tier, headline strengths/gaps |
| **What changed?** | Since prior assessment, project, or review — measurable delta |
| **Why does it matter?** | Business risk, productivity, reliability, continuity in plain language |
| **What should happen next?** | Top priorities, approval asks, or roadmap headline |

Executive summaries are **one to two pages maximum**. No question-level assessment detail in this section.

---

# 7. Section Layout Standards

| Rule | Specification |
| ---- | ------------- |
| Heading hierarchy | H1 document title (cover only) → H2 major sections → H3 subsections — no skipped levels |
| Section order | Business summary → profile/impact → initiatives → detail → technical appendix (if any) → **Technology Journey closing page** (always last) |
| Tables | Zebra striping or row rules; column headers repeated on page breaks |
| Lists | Bullet for opportunities; numbered for phased recommendations |
| Page breaks | Major sections start on new page when content exceeds half page |
| Headers | Client name + document title (abbreviated) on interior pages |
| Footers | Confidential line + page number; StackScore mark optional |

Long tables use **continued** headers and avoid splitting a single row across pages.

# 8. Charts & Visualizations

Standard visual components (styling per DOC-005):

| Component | Use case |
| --------- | -------- |
| **Technology Profile gauge** | Overall StackScore on profile and assessment PDFs |
| **Category radar chart** | Seven v2 category comparison |
| **Historical trend chart** | StackScore / maturity over snapshots |
| **Technology Journey progress** | Current → completed → projected → vision (Section 9) |
| **Business impact summary cards** | Key outcome metrics (risk reduced, score delta) |
| **Roadmap timeline** | Phased initiatives on roadmap and QBR PDFs |
| **Before vs After comparison** | Side-by-side or two-column environment narrative |

**Chart rules:** accessible palette with labels/legends; source footnote with data date; no internal benchmarks on client PDFs unless approved.

# 9. Technology Journey Visualization

All **major client-facing** PDFs (Profile, Assessment, TIP, Roadmap, Completion, QBR) should include the standard Technology Journey graphic:

```text
Current Technology Profile
        ↓
Completed Improvements
        ↓
Projected Future Improvements
        ↓
Long-Term Technology Vision
```

The graphic is **simplified for executives** — numeric detail in adjacent summary tables.

---

# 10. Business Impact Presentation

Reports shall prioritize:

* Business outcomes
* Technology Profile improvements
* StackScore improvements
* Risk reduction
* Productivity improvements
* Technical debt reduction
* Reliability improvements
* Operational maturity

**Order:** business impact summary → profile comparison → category highlights → initiative detail → technical appendix (if any) → Technology Journey closing page.

---

# 11. Technology Journey Closing Page

Every StackScore PDF shall end with a **Technology Journey closing page** — the final content page before any blank end sheet.

This page reinforces continuous improvement and StackScore's role in the client's technology journey. It appears **after** all report-specific content (including type-specific closing statements such as the Completion Report narrative in DOC-107).

**Required elements:**

* Section title: *Your Technology Journey*
* BobKat IT and StackScore branding (logo or mark)
* Standard closing message (exact text unless locale translation is enabled in future)
* Optional: simplified Technology Journey graphic (Section 9) when not shown earlier in the document

**Standard closing message:**

> Every technology investment should move your organization closer to a more secure, reliable, efficient, and resilient operating environment. StackScore exists to measure, guide, and document that journey.

**Applicability:**

| PDF class | Closing page |
| --------- | ------------ |
| Client-facing strategic PDFs | **Required** |
| Client-facing operational PDFs | **Required** |
| Internal consulting PDFs | **Required** (same message; internal footer may apply) |
| Internal admin/financial PDFs | **Recommended** |

The closing page uses the same typography and brand colors as the cover. It must not introduce new data, pricing, or technical detail.

---

# 12. Technical Appendix Rules

* Technical detail appears **after** business summaries.
* **Supplemental Technical Assessment Report** is the primary vehicle for question-level content.
* Primary client PDFs may reference supplemental appendix — not embedded by default.
* Appendix pages use header: *Technical Appendix*.
* Internal notes and evidence never in client appendix unless explicitly released.

---

# 13. Confidentiality Rules

Client-facing PDFs shall **never** expose:

* Internal costs
* Vendor pricing
* Margins
* Pricing formulas
* Solution Playbook names
* Internal consultant notes
* Administrative comments

PDF generation must apply DOC-125 redaction rules before render. TIP PDFs show investment totals only — no line-item cost breakdown unless business-approved.

---

# 14. Versioning & Document Metadata

Embedded PDF metadata: `Title`, `Author` (BobKat IT), `Subject`, `CreationDate`, `Keywords` (clientId, reportType, version).

Visible version block: document version, `GeneratedAt`, `GeneratedBy`, source entity ids in internal metadata only.

**Delivered** client PDFs are immutable — regeneration creates a new version; prior PDF retained.

---

# 15. Export Options

| Option | Specification |
| ------ | ------------- |
| **Download** | Direct download; audit logged for sensitive types |
| **Archive** | Document Service with `documentType` and visibility |
| **Print** | Print-friendly margins |
| **Future email** | Portal link or encrypted attachment per DOC-125 |
| **Batch export** | Admin-only — internal PDF variant |

Default page size: **US Letter**.

---

# 16. Accessibility Standards

Generated PDFs should support:

* Professional typography (minimum 10pt body, 14pt+ headings)
* High contrast (WCAG AA target for text and chart labels)
* Accessible color usage (patterns/labels supplement color)
* Logical tagged heading hierarchy (H1–H3)
* Searchable and selectable text
* Print-friendly margins

Future: PDF/UA compliance for client-facing deliverables.

# Standard PDF Document Types

Each type defines: **Purpose**, **Audience**, **Required Sections**, **Optional Sections**, **Branding Rules**, **Visibility Rules**, **Versioning Requirements**.

## Technology Profile Report

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | PDF export of current client technology posture. |
| **Audience** | Executive / Business Owner, Client User, Consultant |
| **Required sections** | Cover, executive summary, profile gauge, category radar, risk summary, opportunities, Technology Journey graphic, Technology Journey closing page |
| **Optional sections** | Trend sparkline, managed program callout |
| **Branding rules** | Full client-facing standard branding |
| **Visibility rules** | Client-safe; no internal notes |
| **Versioning** | `generatedAt` stamp; living export unless snapshotted |

## StackScore Assessment Report

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | PDF of completed assessment results in business language. |
| **Audience** | Executive / Business Owner, Client User (redacted), Consultant |
| **Required sections** | Cover, executive summary (four questions), overall score, category breakdown, top risks/opportunities, Technology Journey closing page |
| **Optional sections** | Prior assessment comparison, Technology Journey, supplemental appendix link |
| **Branding rules** | Full client-facing |
| **Visibility rules** | Client variant excludes internal notes and evidence |
| **Versioning** | One PDF per completed assessment — immutable |

## Technology Improvement Plan

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Client-facing strategic improvement proposal PDF. |
| **Audience** | Executive / Business Owner, Client User, Consultant |
| **Required sections** | Cover, executive summary, recommendations, phased priorities, investment summary, approval/signature block, Technology Journey closing page |
| **Optional sections** | Profile projection, business outcomes, Technology Journey |
| **Branding rules** | Full client-facing; branded signature area |
| **Visibility rules** | No margins, playbook names, or labor rates |
| **Versioning** | **Required** — version on cover; superseded PDFs archived |

## Technology Roadmap

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Phased improvement plan PDF. |
| **Audience** | Executive / Business Owner, Client User, Consultant |
| **Required sections** | Cover, executive summary, roadmap timeline, phase objectives, projected profile progression, Technology Journey closing page |
| **Optional sections** | Phase budget totals (client-visible), Technology Journey |
| **Branding rules** | Full client-facing |
| **Visibility rules** | No internal dependency codes |
| **Versioning** | **Required** on structural change |

## Technology Completion Report

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Post-project business value deliverable. |
| **Audience** | Executive / Business Owner, Client User, Consultant |
| **Required sections** | Per DOC-107: cover, executive summary, business impact, profile before/after, completed initiatives, before/after, warranty, next steps, closing statement, Technology Journey closing page |
| **Optional sections** | Business metrics cards, Technology Journey |
| **Branding rules** | Full client-facing |
| **Visibility rules** | Fully client-safe |
| **Versioning** | Immutable after delivery |

## Quarterly Business Review

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Periodic executive review PDF. |
| **Audience** | Executive / Business Owner, Consultant, Read Only User |
| **Required sections** | Cover, current profile, score trend, completed work, remaining recommendations, roadmap progress, upcoming priorities, managed program status, Business Impact Summary, Technology Journey closing page |
| **Optional sections** | Multi-quarter Technology Journey, initiative deep-dives |
| **Branding rules** | Client presentation variant |
| **Visibility rules** | Client variant excludes consultant prep notes |
| **Versioning** | One PDF per review cycle — archived |

## Business Impact Summary

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Standalone or embedded before/after improvement narrative. |
| **Audience** | Executive / Business Owner, Consultant |
| **Required sections** | Previous vs updated profile, StackScore delta, category improvements, risks reduced, business benefits, completed initiatives, next steps, Technology Journey closing page |
| **Optional sections** | Expected vs actual variance chart |
| **Branding rules** | Client-facing when delivered |
| **Visibility rules** | Client-safe |
| **Versioning** | Tied to parent report or `generatedAt` for standalone |

## Recommendation Summary

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Internal prioritized opportunity list PDF. |
| **Audience** | Consultant, Internal Admin |
| **Required sections** | Cover (internal), priority table, status, category, investment estimate, linked project, Technology Journey closing page |
| **Optional sections** | Playbook mapping, consolidation view |
| **Branding rules** | Internal cover banner |
| **Visibility rules** | Internal — playbook names permitted |
| **Versioning** | Point-in-time export |

## Project Summary

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Project status and scope PDF. |
| **Audience** | Consultant, Client User (summary), Technician (assigned) |
| **Required sections** | Cover, objectives, status, timeline, completion progress, Technology Journey closing page |
| **Optional sections** | Task list (role-filtered), expected impact |
| **Branding rules** | Client-facing variant for distribution; simplified for technician |
| **Visibility rules** | No pricing; technician — assigned project only |
| **Versioning** | Live export timestamp |

## Supplemental Technical Assessment Report

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Question-level technical appendix PDF. |
| **Audience** | Consultant, Internal Admin |
| **Required sections** | Cover (technical appendix label), assessment metadata, category-grouped responses, scores, Technology Journey closing page |
| **Optional sections** | Evidence notes, prior assessment comparison |
| **Branding rules** | Internal banner; StackScore mark |
| **Visibility rules** | Not client-facing by default |
| **Versioning** | Immutable per assessment |

## Internal Proposal Package (Admin Only)

| Attribute | Specification |
| --------- | ------------- |
| **Purpose** | Full internal proposal workbook with financial detail. |
| **Audience** | Internal Admin (delegated Consultant subsets) |
| **Required sections** | Cover (internal), TIP content, line-item pricing, labor/cost/margin, proposal health, playbook mapping, Technology Journey closing page (recommended) |
| **Optional sections** | Profitability preview, catalog references |
| **Branding rules** | Internal only — prominent confidentiality |
| **Visibility rules** | Admin only — never client or technician |
| **Versioning** | Synced with TIP version; audit logged on export |
